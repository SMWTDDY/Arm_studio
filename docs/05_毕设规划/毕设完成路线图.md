# 毕设完成路线图：统一遥操、仿真控制与 3DGS 数据增强

本文档根据两份阶段文档和本地 `3DGS` 代码库整理，目标是把当前 `Arm_studio` 合并工程继续收敛成可以支撑毕设答辩的系统方案。

## 一句话定位

毕设不要再扩成“从零训练一个完整 VLA 大模型”。更稳妥的主线是：

```text
低成本真实 Piper 单臂/双臂平台  
+ ManiSkill/SAPIEN 统一仿真控制  
+ 3DGS 真实背景/资产增强   
+ 统一真实/仿真数据协议
+ 轻量策略训练与少样本适配验证
```

论文可以聚焦为：

```text
面向机械臂任务的低成本可适应 VLA/模仿学习系统研究与工业应用探索
```

这里的“VLA”建议写成广义视觉-语言-动作系统或视觉条件动作策略系统，不必把创新点押在训练大模型本身。工程创新点应放在真实/仿真统一、3DGS 辅助数据生成、低成本少样本适配和工业任务验证上。

## 当前材料给出的判断

### 进度复盘 PDF 的核心结论

原始目标是开源 VLA 轻量化部署、本地机械臂任务适配、真实/虚拟工业工位探索。当前偏离点主要在于同时尝试了 SAPIEN、RL、diffusion、动力学、自动生成仿真遥操库等过多方向。

后续应收敛到一条可答辩主线：

```text
实机部署 + 仿真环境模拟部署 + 数据轻量化
```

任务选择也要避开高风险方向。螺丝拧紧、复杂装配、完整流水线都太重；更合适的是桌面分拣、平面推移、简单抓放、抽屉/铰链物体操作这类真实和仿真都能复现的任务。

### SAPIEN/3DGS PDF 的核心结论

RoboSimGS 的公开代码展示的是：

```text
真实采集 -> 3DGS 背景 -> 物体 mesh/URDF/物理参数 -> 仿真合成数据
```

但当前本地 `3DGS` 代码运行时基于 Genesis 和 LeRobot，不是 SAPIEN/ManiSkill。相机位姿对齐代码也没有完整公开。因此它不适合直接整体并入 `Arm_studio`，更适合作为资产生成、世界坐标对齐和高保真背景渲染的参考层。

最合适的架构是两层：

```text
资产层：3DGS 背景、物体 mesh、URDF、物理参数、坐标对齐
仿真层：ManiSkill/SAPIEN 负责物理、碰撞、机器人控制、相机和数据导出
```

### RoboSimGS 论文的新增落地要求

根据 `High-Fidelity Simulated Data Generation for Real-World Zero-Shot Robotic Manipulation Learning with Gaussian Splatting`，后续毕设规划新增一条专项路线：

```text
外部模型导入
静态 3DGS 背景生成
物理属性和运动学结构 AI 自动估计
虚拟环境和现实相机校准
```

具体工程拆解、可复用 3DGS 代码、目标目录结构和验收标准见：

```text
docs/05_毕设规划/RoboSimGS论文功能落地计划.md
```

## 本地 3DGS 代码可复用范围

本地 `3DGS` 库的可用性判断如下：

```text
3DGS/README.md
```

说明该项目发布了 3D 重建、关节/物理估计、世界坐标对齐和 Genesis 数据生成，但 Camera Pose Alignment 仍未释放。

```text
3DGS/DataGeneration/base_task.py
3DGS/DataGeneration/pick_banana.py
```

可作为“如何把 3DGS 背景和物体随机化接进数据生成”的参考，但强绑定 Genesis、LeRobot 和固定路径，不建议直接迁入 `Arm_studio`。

```text
3DGS/Gaussians/render.py
```

可作为 Gaussian 渲染器原型，但存在硬编码路径和 CUDA rasterizer 绑定。后续如果要嵌入工程，必须先重构为读取配置的独立 renderer。

```text
3DGS/utils/icp.py
```

可作为世界坐标对齐原型，但当前参数说明和代码不一致，`size` 没有转成 float，也没有保存 alignment JSON。后续应移植并修复为 `Arm_studio` 的对齐工具。

```text
3DGS/Articulation/
```

可作为物体分割、URDF 生成、物理参数估计的参考。答辩阶段建议只做刚体或准刚体物体；抽屉/铰链类关节物体作为加分项，不作为最低完成目标。

## 推荐毕业任务

首选任务：

```text
桌面物体分拣/归类：把真实桌面上的若干物体抓取或推到指定区域
```

原因：

1. 真实 Piper 能做，仿真也能做。
2. 单臂版本可以先闭环，双臂版本可以扩展为左右臂分别处理左右区域或协同搬运。
3. 3DGS 背景很适合桌面场景，物体 mesh 获取难度低。
4. 数据格式能自然覆盖真实采集、仿真采集、回放、训练和评估。
5. 工业解释可以写成低成本桌面工位分拣、轻量产线前处理或小件归类。

备选扩展任务：

```text
抽屉/盒盖/简单铰链物体操作
```

它更能体现 SAPIEN/URDF 和关节建模价值，但工程风险更高，应作为第二阶段或答辩展示增强项。

## 总体技术路线

最终系统建议拆成六个模块：

```text
统一环境入口
真实/仿真统一数据协议
共享场景双臂仿真
3DGS 资产与渲染桥
自动/半自动数据生成
轻量策略训练与部署评估
```

数据流如下：

```text
真实相机采集
  -> COLMAP / Nerfstudio / 3DGS 重建
  -> 背景 Gaussian PLY + 物体 mesh/URDF + physics.json
  -> ICP/人工校准得到 world_alignment.json
  -> ManiSkill/SAPIEN 加载机器人、物体、相机
  -> 3DGS 背景渲染 + SAPIEN 前景/深度合成
  -> 输出统一 H5/LeRobot 数据
  -> Diffusion/VLA-style 策略训练
  -> 真实 Piper 或仿真环境部署
```

## 阶段计划

### 阶段 1：继续完成统一遥操和仿真控制

目标：

```text
同一个入口通过 config 切换 real/sim、single/dual、joint/pose。
```

要完成：

1. 实现真正共享物理世界的双臂 ManiSkill/SAPIEN 场景，不再只是两个单臂场景拼接。
2. 统一录制脚本，避免旧脚本直接调用 `SinglePiperEnv`、`DualPiperEnv` 或旧 ManiSkill 入口。
3. 让真实主臂控制从臂和控制仿真手臂走同一套 action/obs 协议。
4. 对 `unified_real_single.yaml`、`unified_real_dual.yaml`、`unified_sim_single.yaml`、`unified_sim_dual.yaml` 做一次 round-trip 检查。

验收标准：

```text
real single dry-run 通过
real dual dry-run 通过
sim single 可 reset/step/record
sim dual 在同一场景中可 reset/step/record
真实和仿真的 obs/action key 完全一致
```

### 阶段 2：完成统一数据协议与旧数据转换

目标：

```text
真实采集、仿真采集、旧 Arm_studio 数据、LeRobot 数据都能转成统一 raw H5。
```

要完成：

1. 明确最终 H5 结构：`obs/...`、`action/...`、`meta/env_meta`。
2. 补旧 Arm_studio HDF5 转换器，把 `action + observation/...` 转成统一结构。
3. 统一采集命令，只改 config 即可切换真实/仿真、单臂/双臂。
4. 统一视频检查、shape 检查、回放检查脚本。

验收标准：

```text
同一训练 loader 能读取 real_single、real_dual、sim_single、sim_dual 数据
同一 replay 脚本能回放真实采集和仿真采集
H5 中保存 env_meta，能追溯 backend、arm_mode、control_mode、camera_keys、action_keys
```

### 阶段 3：建立 3DGS 最小可用资产链路

目标：

```text
不是先追求全自动，而是先做出一套可复现实验资产。
```

要完成：

1. 选择一个固定桌面场景，采集多视角图片。
2. 训练或生成桌面背景 Gaussian PLY。
3. 获取 2 到 4 个任务物体 mesh，优先使用简单刚体。
4. 建立 `assets/realsim/<scene_name>/` 资产规范。
5. 从 `3DGS/utils/icp.py` 迁移并修复世界坐标对齐工具，输出 `world_alignment.json`。
6. 相机位姿先采用人工标定或半自动标定，不等待 RoboSimGS 未公开代码。

建议资产目录：

```text
assets/realsim/desktop_sorting/
  background/scene_gaussian.ply
  objects/block_a/mesh.obj
  objects/block_b/mesh.obj
  objects/tray/mesh.obj
  physics.json
  cameras.json
  world_alignment.json
  README.md
```

验收标准：

```text
能从配置读取场景资产
能把真实桌面背景和仿真世界坐标大致对齐
能在文档中说明每个资产来源、尺度和坐标系
```

### 阶段 4：实现 SAPIEN/ManiSkill + 3DGS 渲染桥

目标：

```text
物理交给 SAPIEN/ManiSkill，视觉背景交给 3DGS。
```

要完成：

1. 新增 `gs_bridge` 或 `sim_assets` 模块，封装 Gaussian PLY 加载和相机参数输入。
2. 重构 `3DGS/Gaussians/render.py` 思路，去掉硬编码路径。
3. 从 ManiSkill/SAPIEN 相机导出 intrinsics/extrinsics，传给 Gaussian renderer。
4. 实现最小合成：3DGS 背景 + SAPIEN 前景 RGB。
5. 如果时间允许，再加入深度/alpha 合成，处理遮挡关系。

验收标准：

```text
同一个 sim step 能输出普通 SAPIEN 图像和 GS-composited 图像
H5 中能记录 rgb/base_camera 或 rgb/base_camera_gs
相机轻微扰动后背景仍能稳定渲染
```

注意事项：

```text
不要把 3DGS 当成物理对象。
物体碰撞和抓取必须使用 mesh/URDF。
3DGS 只负责视觉背景或视觉增强。
```

### 阶段 5：自动/半自动生成训练数据

目标：

```text
用较少真实数据 + 较多仿真/GS 增强数据提升策略适配能力。
```

要完成：

1. 桌面物体随机初始位姿。
2. 相机轻微扰动。
3. 光照、材质、纹理随机化。
4. 背景 Gaussian 或背景图像扰动。
5. 单臂示教数据生成。
6. 双臂示教数据生成。
7. 输出统一 H5，并可转 LeRobot。

验收标准：

```text
至少生成 sim-only 数据集
至少生成 GS+sim 数据集
能和少量 real 数据合并训练
```

### 阶段 6：训练、部署与论文实验

目标：

```text
把论文实验设计成可解释、可复现、可答辩，而不是追求最大模型。
```

推荐实验组：

1. Real-only：只用少量真实示教训练。
2. Sim-only：只用普通仿真数据训练。
3. Sim+Domain Randomization：普通仿真加随机化。
4. GS+Sim：3DGS 背景合成仿真数据。
5. GS+Sim+Few Real：3DGS 合成数据加少量真实微调。

指标：

```text
任务成功率
平均完成时间
所需真实示教数量
微调时间
真实/仿真视觉差异
失败类型统计
```

最低可答辩结果：

```text
一个真实任务能跑通
一个同构仿真任务能跑通
同一套数据协议贯穿真实/仿真
GS+Sim 至少展示出视觉增强和数据生成能力
策略训练能给出 baseline 对比
```

## 近期两周执行清单

第一优先级：

1. 完成真实共享场景双臂 ManiSkill/SAPIEN 环境。
2. 统一采集脚本，让 config 成为唯一入口。
3. 补旧 HDF5 到统一 raw H5 的转换器。
4. 选定毕业主任务：建议桌面分拣/归类。

第二优先级：

1. 建立 `assets/realsim/desktop_sorting/` 资产规范。
2. 修复并迁移 ICP 对齐脚本，输出 JSON。
3. 跑通一个 3DGS 背景 PLY 的离线渲染。
4. 让仿真相机参数能喂给 3DGS renderer。

第三优先级：

1. 实现 GS 背景 + SAPIEN 前景合成。
2. 生成第一批 GS+Sim H5 数据。
3. 接入 diffusion 训练，先不合并复杂 VLA。
4. 设计真实部署评估表格。

## 需要避免的坑

1. 不要把毕业目标写成“完整工业流水线系统”，应写成“工业应用探索与可行性验证”。
2. 不要一开始追求全自动关节物体建模，刚体分拣更稳。
3. 不要把 `3DGS/DataGeneration` 直接迁进来，它绑定 Genesis 和固定路径。
4. 不要依赖 RoboSimGS 未发布的 Camera Pose Alignment，先采用人工/半自动标定。
5. 不要把训练创新点压得过重，当前系统工程创新更容易完成和解释。
6. 不要让真实数据和仿真数据分裂成两套脚本，否则后期训练和论文实验会失控。

## 与当前合并状态的关系

当前 `docs/01_项目总览/合并状态说明.md` 中列出的未完成工作，正好对应本文档的前两个阶段。建议优先完成：

```text
共享场景双臂仿真
统一 config 采集脚本
旧 HDF5 转换器
真实/仿真 round-trip 验证
```

完成后再进入 3DGS/SAPIEN 高保真数据增强。这样工程主线不会散，论文叙事也会更稳。

## 最终答辩叙事建议

答辩时可以按下面逻辑讲：

1. 低成本机械臂平台很难直接依赖大规模真实数据。
2. 本项目先统一了真实 Piper 和 ManiSkill/SAPIEN 仿真的控制/数据协议。
3. 在此基础上，利用 3DGS 复原真实桌面背景，用 mesh/URDF 表达可交互物体。
4. 仿真中自动生成与真实格式一致的数据，减少真实示教成本。
5. 通过少量真实数据微调策略，实现从仿真到真实的任务适配。
6. 以桌面分拣/归类任务验证系统可行性，并分析真实部署风险。

这条叙事既覆盖 VLA/策略学习，也覆盖 SAPIEN/3DGS/真实机械臂，不会被某一个未完全成熟的模块拖死。
